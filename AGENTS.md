# Circuit Hero 项目规则

## 项目定位

Circuit Hero 是一个面向 12-16 岁学习者的电路学习互动原型。当前项目保持静态网站形态，优先完成既有 PRD / 原型中已经出现的学习流程，不主动新增视觉稿中没有的页面或功能。

## 目录约定

- `index.html`：首页入口。
- `login.html`：登录 / 注册原型入口。
- `story-map.html`：章节地图。
- `mission-briefing.html`：关卡任务介绍。
- `workbench.html`：电路搭建工作台。
- `css/`：页面样式与通用设计变量。
- `js/data/`：关卡、进度、后端前置数据模型等纯数据与状态逻辑。
- `js/pages/`：各 HTML 页面的入口脚本。
- `js/game/`：工作台、电路判断、拖拽、连线、反馈等核心玩法逻辑。
- `js/components/`：跨页面或跨玩法的轻量组件逻辑。
- `assets/`：图片、图标等静态资源。
- `tests/`：零依赖验证脚本。新增可抽离纯逻辑时优先补测试。
- `docs/`：产品、数据结构、后端接口等说明文档。

## 实现原则

- 不新增原型稿没有的视觉页面；占位入口可以禁用或指向已有流程。
- 优先保持纯静态可运行；引入后端前先明确数据模型和 API 边界。
- UI 文案与页面结构尽量复用现有风格，不做无关视觉重设计。
- 关卡状态、评分、计时、提示使用情况必须按关卡隔离，不能串关。
- 业务逻辑尽量放在 `js/data/` 或可测试模块中，页面脚本只做 DOM 绑定。
- 不把密码、密钥、token、生产配置写入仓库。

## 验证方式

改动后至少执行：

```powershell
npm test
```

如果修改了页面交互，还需要在浏览器中手动验证：

- 首页能进入登录页或故事地图。
- 地图不会跳转到不存在的关卡。
- 任务介绍页能进入工作台。
- 工作台完成关卡后能保存对应关卡的星级与完成状态。

## 后端接入边界

登录注册后端建议后续使用 `Node.js + Express + SQLite`。接入前先完成：

- 用户表结构。
- 关卡进度表结构。
- 本地进度对象与后端 API 的字段映射。
- 前端从 `localStorage` 迁移到 API 的兼容策略。

涉及 `.env`、数据库迁移、部署、生产密钥时必须先确认。
